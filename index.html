DevSecOps Architecture for CAASM Platform

Security-first DevSecOps platform for a Cyber Asset Attack Surface Management application on on-prem Kubernetes with GitOps-driven delivery and full observability.

  • Kubernetes · GitOps · ArgoCD
  • Spring Boot · Angular · PostgreSQL
  • SonarQube · Trivy · Keycloak · New Relic
📋 Overview

Designed a security-first DevSecOps platform for a Cyber Asset Attack Surface Management (CAASM) application running on an on-prem Kubernetes cluster.

The platform automated the full delivery lifecycle using GitHub Actions for CI and ArgoCD for GitOps-based deployment, ensuring cluster state always matched Git configuration.

The application stack consisted of an Angular frontend and a Spring Boot backend, with PostgreSQL for transactional data and Elasticsearch powering large-scale search and analytics across cyber asset records.

🔧 DevSecOps Pipeline

A comprehensive CI pipeline was implemented using GitHub Actions to build, test, and secure application artifacts. Security and quality gates integrated directly into the pipeline:

  • SonarQube — Static code analysis and quality enforcement
  • Trivy — Container vulnerability scanning
  • Nexus Repository — Centralized artifact and image management

This shift-left approach ensured that vulnerable builds were blocked before reaching the runtime environment.

🚀 Deployment Architecture

Continuous delivery was implemented using ArgoCD and the GitOps model, enabling:

  • Declarative Kubernetes deployments
  • Automatic drift detection
  • Fully traceable releases through Git history

The application ran on an on-prem Kubernetes cluster behind an ingress controller. Authentication and identity management were handled through Keycloak using OIDC, providing centralized SSO for the platform.

📊 Observability

Application performance and reliability were monitored using New Relic APM, allowing the team to track:

  • Request latency and error rates
  • Transaction traces across services
  • Performance regressions after releases

This significantly improved operational visibility and incident response capabilities.

📈 Impact
  • Reduced deployment cycle from manual multi-step releases to automated GitOps deployments
  • Introduced security gates in CI to block vulnerable container images
  • Standardized artifact management through Nexus
  • Improved observability and release confidence with APM monitoring
  • Enabled scalable search capabilities through PostgreSQL + Elasticsearch hybrid architecture

End-to-End AWS EKS GitOps Platform

Cloud-native DevOps platform modernizing a legacy EC2-based PHP Symfony infrastructure to AWS EKS with GitOps, ephemeral environments, and full observability.

  • Terraform · Ansible · EKS
  • Jenkins · ArgoCD · Helm
  • MariaDB · RabbitMQ · Redis
📋 Overview

Designed and implemented a cloud-native DevOps platform on AWS to modernize a legacy EC2-based PHP Symfony infrastructure.

The original architecture relied on EC2 auto-scaling groups, a single MySQL database, and AWS CodePipeline/CodeBuild. This setup caused high infrastructure costs and database bottlenecks during traffic spikes generated by periodic marketing campaigns (car lottery).

The platform was completely redesigned using Terraform-managed infrastructure, an EKS Kubernetes cluster, Jenkins CI with Kubernetes agent pods, ArgoCD GitOps deployments, and a comprehensive observability stack.

🏗️ Infrastructure & Configuration

All infrastructure provisioned with Terraform, executed via Jenkins pipelines:

  • VPC, Subnets, Security Groups, ALB
  • EKS Cluster with managed Node Groups
  • ECR, IAM Roles, Jenkins Infrastructure
  • Tailscale VPN for secure cluster access
  • Microsoft Entra ID for identity management

Ansible used for Jenkins configuration and cluster bootstrap operations.

🔧 CI/CD & GitOps Pipeline

CI — Jenkins with Groovy/DSL pipelines running on Kubernetes agent pods:

  • SonarQube — Static code analysis and quality gates
  • Trivy — Container vulnerability scanning
  • Docker Build → Push to AWS ECR

CD — ArgoCD (GitOps) with Helm charts:

  • GitHub Actions syncs repository changes with ArgoCD
  • Declarative deployments via Helm charts to EKS
🧪 Ephemeral Dev Environments

When a developer opens a PR, Jenkins dynamically creates an isolated environment:

  • ALB, Application pods, Database, and RabbitMQ — all provisioned via Terraform
  • Developer tests in complete isolation before merge
  • After tests pass: Terraform Destroy cleans up all resources

Production environment runs persistent infrastructure with destroy pipeline disabled — only DevOps team access.

📊 Observability & Operations

Three-layer observability stack:

  • Prometheus + Grafana — Node, pod, MySQL, and RabbitMQ metrics
  • New Relic APM — PHP application monitoring and error tracking
  • Fluentd — Centralized log collection from all containers

Velero for Kubernetes resource and persistent volume backups/DR.

MariaDB Operator deployed instead of RDS (client declined due to cost) — providing replication and high availability in-cluster. Redis used for query caching and session storage.

📈 Impact
  • Eliminated EC2 infrastructure costs with containerized workloads on EKS
  • Resolved database bottlenecks through RabbitMQ async processing during campaign spikes
  • Ephemeral environments enabled isolated developer testing per PR
  • GitOps deployments replaced manual releases with full audit trail
  • Three-layer observability replaced CloudWatch-only monitoring
  • Velero backups provided disaster recovery for all Kubernetes resources

High-Availability Laravel News Platform on AWS EKS

Auto-Scaling Kubernetes Architecture for Breaking-News Traffic

Migrated a legacy EC2-based Laravel news platform to a scalable containerized architecture on AWS EKS with automated scaling, Redis caching, and Cloudflare CDN.

  • Laravel · AWS EKS · Terraform
  • Redis · RDS · S3 · Cloudflare
  • Helm · CodePipeline · HPA · Prometheus
📋 Overview

Modernized a high-traffic news platform that experiences unpredictable traffic spikes during breaking news events.

The original infrastructure relied on EC2 auto-scaling groups, a single MySQL database, CloudFront CDN, and AWS CodePipeline/CodeBuild/CodeDeploy. While it worked under normal conditions, it struggled during sudden traffic surges — leading to scaling delays, database bottlenecks, and high infrastructure costs.

The platform was redesigned using a containerized architecture on AWS EKS with Terraform, automated dual-layer scaling, Redis caching, Cloudflare CDN, and Prometheus/Grafana observability.

🏗️ Infrastructure

All infrastructure provisioned with Terraform:

  • AWS VPC with public/private subnets
  • EKS Cluster with managed Node Groups
  • Security Groups, IAM Roles
  • Application Load Balancer + Kubernetes Ingress
  • Amazon RDS (MySQL) for persistent data
  • Amazon S3 for media storage (images/videos)

Cloudflare provides CDN, DNS management, and DDoS protection. Media assets served from S3 through Cloudflare globally.

🔧 Deployment Workflow

AWS-native CI/CD pipeline with Helm-based deployment:

  • Developer push → GitHub repository
  • CodePipeline triggered automatically
  • CodeBuild — Runs tests and builds Docker image
  • Push image → Amazon ECR (container registry)
  • Helm upgrade — Updates image tag in Helm chart values
  • Rolling Deployment — New pods created alongside old pods
  • Readiness checks pass — Health probes validate new pods
  • Traffic shifts — Old pods terminated, traffic routes to new pods
⚡ Scaling Strategy

Dual-layer auto-scaling for handling breaking-news traffic:

  • HPA (Horizontal Pod Autoscaler) — Scales Laravel pods based on CPU/memory metrics
  • Cluster Autoscaler — Provisions/removes EC2 worker nodes based on pod scheduling demand
  • PDB (Pod Disruption Budgets) — Maintains minimum availability during scaling events
  • Health Probes — Readiness/liveness probes ensure only healthy containers receive traffic
💾 Redis Caching Layer

Redis reduces database pressure significantly during traffic spikes:

  • Homepage content caching
  • Breaking news widget caching
  • Popular article metadata
  • Session storage
  • Rate limiting
📊 Observability

Prometheus collects container, node, scaling event, and resource utilization metrics.

Grafana dashboards visualize cluster health, application performance, and auto-scaling activity — enabling engineers to quickly identify bottlenecks during traffic spikes.

📈 Impact
  • Zero downtime during breaking-news traffic spikes — pods and nodes scale automatically
  • Redis caching significantly reduced repeated database queries
  • Media assets served globally through S3 + Cloudflare CDN
  • Cluster Autoscaler enables scale-down during low traffic, reducing costs
  • Prometheus + Grafana provide full visibility into scaling behavior
  • Rolling deployments ensure minimal service disruption

Laravel (PHP) AWS EKS Terraform Redis Amazon RDS Amazon S3 Cloudflare CDN AWS CodePipeline AWS CodeBuild AWS ECR Kubernetes HPA Cluster Autoscaler Prometheus Grafana Helm Docker

Production-Grade Django Startup Platform on AWS ECS Fargate

Serverless Containers · Blue/Green Deployment · CloudWatch Observability

Designed a production-grade Django platform for a startup using ECS Fargate with blue/green deployment, WAF security, CloudWatch alerting, and operational runbooks.

  • ECS Fargate · Terraform · RDS MySQL
  • Blue/Green · WAF · Secrets Manager
  • GitHub Actions · CloudWatch · SNS
📋 Overview

Designed a production-grade Django platform for a fast-growing startup building a subscription-based digital content platform.

The startup needed to launch fast with minimal ops overhead — no server management, zero-downtime deployments, strong security, automated backups, and centralized alerting for a small engineering team.

ECS Fargate was chosen as the runtime because it eliminates EC2 management, patching, and cluster node overhead — making it ideal for a startup that needs to move quickly with a lean team.

🏗️ Infrastructure

All infrastructure provisioned with Terraform:

  • VPC with public/private subnets, Internet Gateway, NAT Gateway
  • ECS Fargate Cluster + Service
  • Application Load Balancer + Target Groups (blue/green)
  • Amazon RDS MySQL (automated backups, 7-day retention, storage autoscaling)
  • Amazon ECR for container images
  • S3 bucket for static/media assets
  • AWS WAF for edge protection
  • AWS Secrets Manager for Django secret key and DB credentials
  • CloudWatch Log Groups + SNS Topic
  • IAM Roles and Policies (task-level access control)
🔧 Deployment Workflow

GitHub Actions-based CI/CD pipeline:

  • Developer push → GitHub repository
  • GitHub Actions — Run tests
  • Docker build — Build application image
  • Push image → Amazon ECR
  • Trigger ECS blue/green deployment
🟦🟩 Blue/Green Deployment

Zero-downtime deployment strategy for production safety:

  • Blue — Current production version serving live traffic
  • Green — New release deployed alongside blue
  • Health checks validate the green task set
  • ALB shifts traffic from blue to green
  • If issues detected → instant rollback to blue
  • Blue tasks terminated after successful cutover

"We used blue/green deployment on ECS to ensure zero-downtime releases and safer rollback during production updates."

🛡️ Security Design
  • AWS WAF — Placed in front of ALB for bot protection, rate limiting, and web exploit filtering
  • Secrets Manager — Django secret key and DB credentials injected at runtime, never hardcoded
  • IAM Task Roles — Each ECS task accesses only the AWS services it needs
  • Security Groups — Network-level isolation between components
💾 Backup & Data Protection
  • RDS automated backups enabled with 7-day retention
  • Point-in-time restore support
  • Automatic snapshot lifecycle management
  • Storage autoscaling — DB grows without manual resize
📊 Observability & Alerting

CloudWatch monitors ECS task logs, ALB metrics, ECS CPU/memory, RDS CPU/connections/storage, and deployment events.

Alarm examples: ECS CPU high, task unhealthy, ALB 5xx errors, RDS CPU high, low free storage, deployment failure.

Alerting flow: CloudWatch Alarms → SNS → Slack + Email notifications. The team is instantly notified of scaling events, errors, and deployment issues.

📋 Runbooks & Documentation

Runbooks prepared:

  • Deployment rollback runbook
  • ECS task unhealthy runbook
  • RDS high CPU runbook
  • ALB 5xx troubleshooting runbook
  • WAF false positive runbook

Documentation: Architecture overview, environment variables, deployment workflow, incident escalation path, backup/restore notes, access/IAM guide.

📈 Impact
  • No server management overhead — fully serverless with ECS Fargate
  • Zero-downtime deployments with blue/green strategy
  • Safer production releases with instant rollback capability
  • Improved security with WAF, Secrets Manager, and IAM isolation
  • Automated operational alerting via CloudWatch + SNS + Slack/Email
  • Simplified database operations with RDS backups and storage autoscaling
  • Better operational readiness with runbooks and platform documentation

Django (Python) Gunicorn AWS ECS Fargate Terraform Amazon RDS MySQL Amazon S3 Amazon ECR AWS WAF AWS Secrets Manager GitHub Actions Blue/Green Deploy CloudWatch Amazon SNS ALB IAM Roles Docker

The solution leverages Google Kubernetes Engine for containerized microservices, Cloud Build + Cloud Deploy for automated delivery pipelines with staging/production promotion, CloudSQL PostgreSQL for transactional data, Google Cloud CDN for global content delivery, and Cloud Trace + Cloud Monitoring for end-to-end observability.

🏗️ Infrastructure

  • GKE Cluster with node auto-provisioning

  • Cloud Storage buckets for product images, media assets, and static files

🔧 CI/CD Pipeline

Google Cloud-native CI/CD with automated delivery pipeline:

  • Developer push → GitHub repository

  • Cloud Deploy — Delivery pipeline with promotion stages

  • Production — Manual approval gate, then promoted to production namespace

📦 GKE Microservices

Containerized microservices on Google Kubernetes Engine:

CloudSQL PostgreSQL — Primary database for:

HA configuration with automated backups, 7-day retention, and point-in-time recovery.

Memorystore for Redis — Used for:

Deployed a production-grade 3-tier application (Frontend + Backend + Database) on AWS EKS, fully provisioned with Terraform. Multi-AZ high availability with ALB ingress and RDS PostgreSQL.

  • EKS · Terraform · VPC
  • React · Node.js · RDS
  • ALB · ECR · GitHub Actions
Details
  • Infrastructure as Code: Terraform provisions complete AWS environment — VPC with public/private subnets across 2 AZs, EKS cluster, node groups, IAM roles, and security groups.
  • Frontend Tier: React/Angular SPA served via Nginx containers with multi-stage Docker builds; routed through ALB Ingress Controller.
  • Backend Tier: Node.js/Python API pods handling business logic; communicates with database via Kubernetes Services and internal DNS.
  • Database Tier: RDS PostgreSQL Multi-AZ for high availability; credentials managed via Kubernetes Secrets and AWS Secrets Manager.
  • CI/CD: GitHub Actions pipeline builds images, pushes to ECR, and deploys via kubectl rolling updates for zero-downtime releases.
  • Impact: Fully reproducible infrastructure; single command to provision or tear down entire stack; 99.9% uptime with multi-AZ resilience.

Blue/Green Deployment on AWS with Jenkins & Terraform

Implemented zero-downtime Blue/Green deployment strategy on AWS using Jenkins CI/CD pipelines and Terraform-managed infrastructure with ALB traffic switching.

  • Jenkins · Terraform
  • ALB · ASG · Route 53
  • Blue/Green · Zero-Downtime
Details
  • Dual Environment: Terraform provisions identical Blue and Green environments (VPC, ASG, ALB Target Groups) for instant rollback capability.
  • Jenkins Pipeline: Multi-stage Jenkinsfile orchestrates build, test, deploy-to-green, smoke-test, and ALB traffic switch with manual approval gate.
  • Traffic Switching: ALB listener rules shift 100% traffic from Blue to Green target group after health checks pass; instant rollback by reverting listener.
  • Infrastructure as Code: All AWS resources (VPC, subnets, ASGs, ALB, Route 53, IAM) defined in Terraform modules for reproducible deployments.
  • Impact: Zero-downtime releases with sub-second rollback; eliminated deployment-related incidents by 95%.
